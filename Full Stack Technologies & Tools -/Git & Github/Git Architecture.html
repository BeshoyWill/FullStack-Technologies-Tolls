Git Architecture

Git Architecture (Brief Explanation)

Git’s architecture is built around a distributed version control system that efficiently tracks changes in code. It’s composed of several core components and concepts:

1. Repository (Repo)

The repository is the heart of Git, where all versions of your project files are stored. It consists of two types:

  • Local repository: Stored on your local machine, where you make changes, commits, and manage versions.
  • Remote repository: Hosted on a server (like GitHub or GitLab) for collaboration with others.

2. Working Directory

This is the directory on your local machine where you actively work on your project. It contains the actual files and folders you’re editing.

3. Staging Area (Index)

The staging area is a temporary space where you gather changes before committing them to the repository. When you use git add, changes move from the working directory to the staging area, allowing you to prepare a commit.

4. Commit

A commit is a snapshot of your project at a particular point in time. It includes the changes staged and a unique identifier (hash) to track it. Commits are like save points that allow you to track and restore previous versions.

5. Branches

Git uses branches to manage different versions of the code. The default branch is usually main or master, and you can create other branches (e.g., for new features or bug fixes) and merge them later.

6. Git Object Model

Git stores data as objects in three main types:

  • Blob: Stores file contents.
  • Tree: Represents a directory and links to blobs and other trees.
  • Commit: Contains metadata (author, date, message) and links to trees, representing the project state at a specific time.

7. HEAD

The HEAD is a reference that points to the current commit or branch you’re working on. It helps Git know what version of the project is active in the working directory.

Git Workflow:

  1. Modify files in the working directory.
  2. Stage changes with git add, moving them to the staging area.
  3. Commit the changes with git commit, creating a snapshot in the local repository.
  4. Push the changes to the remote repository with git push to share them with others.

 

Explaining in Arabic :-

 

شرح معمارية Git

معمارية Git مبنية حول نظام تحكم بالإصدارات الموزعة (Distributed Version Control System) الذي يتابع التغييرات في الشيفرة البرمجية بشكل فعال. وتتكون من عدة مكونات أساسية ومفاهيم:

1. المستودع (Repository)

المستودع هو الأساس في Git، حيث يتم تخزين جميع إصدارات ملفات المشروع. يتكون من نوعين:

  • المستودع المحلي (Local Repository): يتم تخزينه على جهازك المحلي، حيث تقوم بإجراء التعديلات، الالتزامات (commits)، وإدارة الإصدارات.
  • المستودع البعيد (Remote Repository): يتم استضافته على خادم (مثل GitHub أو GitLab) لتسهيل التعاون مع الآخرين.

2. الدليل العامل (Working Directory)

هذا هو المجلد الموجود على جهازك المحلي الذي تعمل فيه على المشروع. يحتوي على الملفات والمجلدات التي تقوم بتحريرها وتعديلها بشكل مباشر.

3. منطقة التحضير (Staging Area)

منطقة التحضير هي مساحة مؤقتة يتم فيها جمع التعديلات قبل الالتزام بها في المستودع. عندما تستخدم الأمر git add، تنتقل التعديلات من الدليل العامل إلى منطقة التحضير، مما يسمح لك بتجهيز التعديلات قبل الالتزام بها.

4. الالتزام (Commit)

الالتزام هو لقطة لحالة المشروع في نقطة زمنية معينة. يتضمن التعديلات التي تم إعدادها في منطقة التحضير ومعرفًا فريدًا (hash) لتتبع التعديل. يُعتبر الالتزام كـ "نقطة حفظ" يمكن العودة إليها لاحقًا.

5. الفروع (Branches)

Git يستخدم الفروع لإدارة النسخ المختلفة من الشيفرة. الفرع الافتراضي يكون عادةً main أو master، ويمكنك إنشاء فروع أخرى (مثلًا للعمل على ميزات جديدة أو لإصلاح الأخطاء) ثم دمجها لاحقًا.

6. نموذج الكائنات في Git (Git Object Model)

Git يخزن البيانات ككائنات بثلاثة أنواع رئيسية:

  • Blob: يخزن محتوى الملفات.
  • Tree: يمثل المجلدات ويربطها بـ blobs وأشجار أخرى.
  • Commit: يحتوي على البيانات الوصفية (مثل المؤلف، التاريخ، الرسالة) ويرتبط بالأشجار التي تمثل حالة المشروع في وقت معين.

7. الرأس (HEAD)

HEAD هو مرجع يشير إلى الالتزام أو الفرع الحالي الذي تعمل عليه. يساعد Git في معرفة أي نسخة من المشروع نشطة في الدليل العامل.

سير العمل في Git:

  1. تعديل الملفات في الدليل العامل.
  2. تحضير التعديلات باستخدام git add، لنقلها إلى منطقة التحضير.
  3. الالتزام بالتعديلات باستخدام git commit، لإنشاء لقطة في المستودع المحلي.
  4. دفع التعديلات إلى المستودع البعيد باستخدام git push لمشاركتها مع الآخرين.

باختصار، تعتمد معمارية Git على المستودع المحلي، الدليل العامل، منطقة التحضير، والالتزامات، وكلها تعمل معًا لمساعدتك على إدارة إصدارات المشروع بشكل فعال. نظام Git الموزع يسمح لعدة مطورين بالعمل في نفس الوقت، حيث يمتلك كل مطور نسخة كاملة من تاريخ المشروع.
